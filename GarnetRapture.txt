방안: frida-gum + LSPlant 통합 → libswappywrapper.so, 단일 SO 자가 부트스트랩 URL 리다이렉트

목표
eversoul-redirect-module(LSPosed + Dobby + XposedInit.java)의 전체 리다이렉트 기능을
기존 libswappywrapper.so 하나로 통합, LSPosed 프레임워크 의존성 제거.
통합 후 SO 하나만 남으며, smali 로딩 방식은 그대로 유지.

  IL2CPP 층(UnityWebRequest.ctor/set_url/SetUrl): frida-gum Interceptor가 Dobby 대체
  Java 층(java.net.URL 3개 생성자 + okhttp3.Request$Builder.url): LSPlant가 LSPosed/Xposed 대체
  LSPlant 콜백 컨테이너: 경량 dex 내장 + InMemoryDexClassLoader + RegisterNatives


1. 신규 third_party 구조 및 빌드 스크립트

third_party/
├── frida-gum/                      # frida-gum devkit 산출물 (수동 1회 빌드 후 commit)
│   ├── include/frida-gum.h         # 통합 단일 헤더
│   └── lib/arm64-v8a/libfrida-gum.a
├── lsplant/                        # LSPlant 소스 (git submodule 또는 clone)
│   ├── lsplant/src/main/jni/lsplant.cc
│   └── lsplant/src/main/jni/include/lsplant.hpp
└── redirect_dex/
    └── RedirectHooks.java          # LSPlant 콜백 컨테이너 (극소 ~6개 정적 메서드)

신규 scripts/build_frida_gum_devkit.sh (수동 1회 실행, 산출물 third_party/frida-gum/ commit):
  NDK r29 = /opt/android-ndk, build.sh와 동일
  cd frida && ./configure --host=android-arm64 --with-devkits=gum && make
  libfrida-gum.a + frida-gum.h → third_party/frida-gum/{lib/arm64-v8a,include}/
  스크립트에 버전 검증 및 안내 메시지 포함

신규 scripts/build_redirect_dex.sh (callback dex 생성, build.sh에 통합 또는 수동 실행):
  javac RedirectHooks.java -d /tmp/redirect_dex_classes
  $ANDROID_HOME/build-tools/*/d8 --release --output /tmp redirect_hooks.dex *.class
  objcopy -I binary -O elf64-littleaarch64 -B aarch64 redirect_hooks.dex redirect_hooks.o
  (기존 PROTO_OBJ와 동일 방식)

LSPlant 소스 취득: git clone https://github.com/LSPosed/LSPlant third_party/lsplant
(플랜 실행 1단계에서 수행)


2. 신규 소스 파일 (src/ 추가)

파일                              역할
src/url_redirect.hpp/.cpp        redirect_url_cpp() 핵심 리다이렉트 로직
                                 (redirect.cpp에서 이관: DOMAINS/WS_DOMAINS/lockincomp/LOCAL_*_BASE 규칙)
                                 순수 함수, 의존성 없음, Java·native 두 계층 hook 공용
src/frida_gum_init.hpp/.cpp      frida-gum 초기화 (gum_init()) 및
                                 gum_inline_attach(target, hook, &orig) 래퍼 유틸
src/native_url_hooks.hpp/.cpp    IL2CPP 층: dlopen libil2cpp.so → il2cpp_* API 파싱 →
                                 UnityWebRequest .ctor/set_url/SetUrl 메서드 포인터 탐색 →
                                 frida-gum Interceptor attach.
                                 detour 내에서 maybe_redirect_il2cpp_string() 호출
                                 (redirect.cpp에서 이관).
                                 별도 스레드에서 libil2cpp 로드 완료 폴링 후 실행
src/lsplant_hooks.hpp/.cpp       Java 층: LSPlant::Init → 내장 redirect_hooks.dex 로드
                                 (InMemoryDexClassLoader) → RegisterNatives로
                                 RedirectHooks 메서드 → native 리다이렉트 로직 바인딩 →
                                 LSPlant::Hook으로 java.net.URL 3개 생성자 +
                                 okhttp3.Request$Builder.url 교체.
                                 okhttp 클래스는 ClassLoader.loadClass hook으로
                                 클래스 로드 시점에 hook (또는 단순 폴링)
src/dex_loader.hpp/.cpp          내장 dex 심볼 읽기 → mmap ByteBuffer 변환 →
                                 InMemoryDexClassLoader로 Class 로드 소형 유틸

RedirectHooks.java (콜백 컨테이너, 극소):

  package com.rikka.eversouloffline;
  public class RedirectHooks {
      public static void urlInit1(String spec) { /* native */ }
      public static void urlInit2(Object ctx, String spec) { /* native */ }
      public static void urlInit3(Object ctx, String spec, Object handler) { /* native */ }
      public static Object okhttpBuilderUrl(Object builder, String url) { /* native */ }
      // 위 4개는 RegisterNatives로 native에 바인딩:
      //   spec/url 읽기 → redirect_url_cpp 호출 → LSPlant backup으로 원 메서드 호출
  }


3. CMakeLists.txt 변경 (ANDROID 분기 한정)

  # add_library(swappywrapper ...) 앞에 추가:

  # frida-gum 정적 라이브러리 (whole-archive: gum 초기화/생성자 반드시 링크)
  add_library(frida_gum STATIC IMPORTED)
  set_target_properties(frida_gum PROPERTIES
      IMPORTED_LOCATION ${CMAKE_CURRENT_SOURCE_DIR}/third_party/frida-gum/lib/arm64-v8a/libfrida-gum.a)
  # devkit 자체 포함 (glib/capstone 이미 병합, _frida_ prefix), 추가 의존 없음

  # LSPlant 소스 직접 swappywrapper에 컴파일
  list(APPEND EVERSOUL_CORE_SOURCES
      src/url_redirect.cpp
      src/frida_gum_init.cpp
      src/native_url_hooks.cpp
      src/lsplant_hooks.cpp
      src/dex_loader.cpp
      third_party/lsplant/lsplant/src/main/jni/lsplant.cc)

  # dex 내장 (PROTO_OBJ와 동일 objcopy 방식)
  set(REDIRECT_DEX_OBJ ...)  # build_redirect_dex.sh 산출물

  add_library(swappywrapper SHARED
      ${EVERSOUL_CORE_SOURCES}
      src/anticheat_patch.cpp
      src/entry.cpp
      ${PROTO_OBJ}
      ${REDIRECT_DEX_OBJ})

  target_include_directories(swappywrapper PRIVATE
      ${EVERSOUL_INCLUDE_DIRS}
      third_party/frida-gum/include
      third_party/lsplant/lsplant/src/main/jni/include)

  target_link_libraries(swappywrapper PRIVATE
      log dl m
      -Wl,--whole-archive frida_gum -Wl,--no-whole-archive)


4. entry.cpp 변경 (초기화 순서)

  __attribute__((constructor)) static void eversoul_entry() {
      eversoul::anticheat::install();          // 불변: 최우선
      eversoul::start_async(kDefaultPort);     // 불변: 백그라운드 서버
      eversoul::frida_gum::init();             // 신규: gum_init(), 경량, constructor 단계 가능
      // Java hook과 IL2CPP hook은 constructor 단계 금지:
      //   이 시점에는 okhttp 클래스 미존재, libil2cpp 미로드, JNIEnv 없음
  }

  JNIEXPORT jint JNICALL JNI_OnLoad(JavaVM* vm, void* reserved) {
      g_vm = vm;                               // 전역 JavaVM 저장, LSPlant::Init 필요
      std::thread([]{
          eversoul::lsplant::install_java_hooks(g_vm);      // LSPlant::Init + URL hook
                                                            // (loadClass 트리거로 okhttp 일괄 처리)
          eversoul::native_hooks::install_il2cpp_hooks();   // libil2cpp dlopen 폴링, frida-gum attach
      }).detach();
      return JNI_VERSION_1_6;
  }

  타이밍 요점:
    constructor 단계: anticheat + server + gum_init (모두 JNIEnv 불필요)
    Java/IL2CPP hook: JNI_OnLoad에서 기동한 스레드 내, ART 완전 준비 후 실행
    java.net.URL: boot class, 로드 시점 빠름 → LSPlant::Init 직후 바로 hook 가능
    okhttp3.*: app class → ClassLoader.loadClass 트리거 또는 단순 폴링

6. 안티치트 유의사항 (구현 시 실측 검증)

  frida-gum은 알려진 메모리 특징 존재 (GumInterceptor 코드 패턴).
  현재 pthread_create hook으로 libcawwyayy 감지 스레드를 무력화했으나,
  LIAPP 외피 본체는 여전히 활성. LIAPP의 gum 특징 스캔 여부 실측 필요.
  감지될 경우 fallback: IL2CPP 층은 기존 anticheat_patch.cpp의 install_inline_hook 재사용
  (추가 의존·특징 없음), Java 층만 LSPlant 사용.

  LSPlant의 ArtMethod 메모리 수정도 특징 존재, 동일하게 실측 필요.

  -fvisibility=hidden 하에서 LSPlant의 [[gnu::visibility("default")]] 심볼과
  swappywrapper 은닉 정책 조율 필요.
  (LSPlant 내부 자기 심볼 호출은 static link 후 문제없음.
   lsplant.cc에 -fvisibility=hidden 미적용 또는
   target_sources PROPERTIES COMPILE_OPTIONS "-fvisibility=default"로 해당 파일만 허용)

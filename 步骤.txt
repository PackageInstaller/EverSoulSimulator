方案：frida-gum + LSPlant 集成进 libswappywrapper.so，单 so 自举完成 URL 重定向
目标
把 eversoul-redirect-module(LSPosed + Dobby + XposedInit.java) 的全部重定向能力折叠进现有的 libswappywrapper.so，去掉对 LSPosed 框架的依赖。集成后只剩一个 so，smali 加载方式不变。

IL2CPP 层(UnityWebRequest.ctor/set_url/SetUrl)：frida-gum Interceptor 替代 Dobby
Java 层(java.net.URL 三个构造函数 + okhttp3.Request$Builder.url)：LSPlant 替代 LSPosed/Xposed
LSPlant callback 容器：嵌入微型 dex + InMemoryDexClassLoader + RegisterNatives

2. 新增源码文件(纳入 src/)
文件	职责
src/url_redirect.hpp/.cpp	redirect_url_cpp() 核心重定向逻辑(从 redirect.cpp 迁移：DOMAINS/WS_DOMAINS/lockincomp/LOCAL_*_BASE 规则)。纯函数,无依赖,被 Java 和 native 两层 hook 共用
src/frida_gum_init.hpp/.cpp	frida-gum 初始化(gum_init())与封装的 gum_inline_attach(target, hook, &orig) 工具函数
src/native_url_hooks.hpp/.cpp	IL2CPP 层：dlopen libil2cpp.so → 解析 il2cpp_* API → 找到 UnityWebRequest 类的 .ctor/set_url/SetUrl 方法指针 → frida-gum Interceptor attach。detour 里调用 maybe_redirect_il2cpp_string()(从 redirect.cpp 迁移)。在独立线程轮询 libil2cpp 加载完成后执行
src/lsplant_hooks.hpp/.cpp	Java 层：LSPlant::Init → 加载嵌入的 redirect_hooks.dex(InMemoryDexClassLoader) → RegisterNatives 绑定 RedirectHooks 的方法到 native redirect 逻辑 → 用 LSPlant::Hook 把 java.net.URL 三个构造函数 + okhttp3.Request$Builder.url 替换成 RedirectHooks 的对应方法。okhttp 类通过 hook ClassLoader.loadClass 一次性触发(类加载时再 hook)
src/dex_loader.hpp/.cpp	从嵌入的 dex 符号读取字节、mmap 成 ByteBuffer、InMemoryDexClassLoader 加载出 Class 的小工具
RedirectHooks.java 极小(callback 容器)：


java
package com.rikka.eversouloffline;
public class RedirectHooks {
    public static void urlInit1(String spec) { /* native */ }
    public static void urlInit2(Object ctx, String spec) { /* native */ }
    public static void urlInit3(Object ctx, String spec, Object handler) { /* native */ }
    public static Object okhttpBuilderUrl(Object builder, String url) { /* native */ }
    // 上面 4 个用 RegisterNatives 绑到 native:
    //   读取 spec/url → 调 redirect_url_cpp → 调 LSPlant backup 原方法
}

    src/dex_loader.cpp

4. entry.cpp 改动(初始化时序)

cpp
__attribute__((constructor)) static void eversoul_entry() {
    eversoul::anticheat::install();          // 不变:必须最先
    eversoul::start_async(kDefaultPort);     // 不变:后台服务端
    eversoul::frida_gum::init();             // 新增:gum_init(),轻量,可 constructor 阶段做
    // Java hook 与 IL2CPP hook 不放 constructor:
    //   此时 okhttp 类不存在、libil2cpp 没加载、且没有 JNIEnv
}

JNIEXPORT jint JNICALL JNI_OnLoad(JavaVM* vm, void* reserved) {
    g_vm = vm;                               // 存全局 JavaVM,LSPlant::Init 需要
    std::thread([]{
        eversoul::lsplant::install_java_hooks(g_vm);  // LSPlant::Init + URL hook(loadClass 一次性触发 okhttp)
        eversoul::native_hooks::install_il2cpp_hooks(); // 轮询 dlopen libil2cpp,frida-gum attach
    }).detach();
    return JNI_VERSION_1_6;
}
时序要点：

constructor 阶段只有 anticheat + server + gum_init(都不需要 JNIEnv)
Java/IL2CPP hook 全部在 JNI_OnLoad 起的线程里,此时有 JNIEnv、ART 完全就绪
java.net.URL 是 boot class,加载早,LSPlant::Init 后可立即 hook
okhttp3.* 是 app class,要在 hook 里加 ClassLoader.loadClass 触发器(类出现时 hook),或简单轮询

6. 反作弊注意事项(实现时验证)
frida-gum 有已知内存特征(GumInterceptor 代码模式)。当前靠 pthread_create hook 废了 libcawwyayy 检测线程,但 LIAPP 外壳本体仍活。需实测 LIAPP 是否扫描 gum 特征。若被检测,fallback 是 IL2CPP 层复用现有 anticheat_patch.cpp 的 install_inline_hook(零新依赖、零特征),只 Java 层用 LSPlant
LSPlant 改 ArtMethod 有内存特征,同理需实测
-fvisibility=hidden 下,LSPlant 的 [[gnu::visibility("default")]] 导出符号与 swappywrapper 的隐藏策略需协调(LSPlant 内部调用自己符号,static link 进来后没问题;确保不给 lsplant 加 -fvisibility=hidden,或用 target_sources ... PROPERTIES COMPILE_OPTIONS "-fvisibility=default" 局部放开 lsplant.cc)

已识别的风险/未决项
frida-gum devkit 实际符号名：需在拿到 frida-gum.h 后确认 gum_init/gum_interceptor_obtain 等的确切声明(prefixed scope 应只影响第三方 glib/capstone,gum 自身 API 保持原名,但需验证)
LSPlant InitInfo 接口版本：不同 LSPlant 版本 InitInfo 字段不同,拿到源码后确认(核心 ART hook 不需 inline_hooker,但 deoptimize boot image 的回调字段以实际版本为准)
dex 嵌入的 d8 路径：依赖 Android SDK build-tools,实现时确认环境可用
